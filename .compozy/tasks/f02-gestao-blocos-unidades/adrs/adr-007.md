# ADR-007: Soft-Delete Padronizado via `ISoftDeletable` + `SoftDeleteableAggregateRoot`

## Status

Accepted

## Date

2026-04-18

## Context

F02 ADR-003 (produto) decidiu que a remoção de blocos e unidades é exclusivamente por inativação (soft-delete) e que esse é o único caminho — nem síndico nem operador podem hard-delete. Essa decisão reverbera nas próximas features do MVP: F03 (moradores também são inativados, nunca deletados), F06 (dispositivos da portaria são revogados, não apagados). F02 é a primeira feature a codificar esse padrão; o desenho técnico escolhido vai virar baseline para todo o projeto .NET.

As opções de implementação divergem em quanto de abstração se introduz:

1. **Marker interface + classe base** que centraliza a lógica (campos `Ativo`, `InativadoEm`, `InativadoPor`, método `Inativar`/`Reativar`) e dispara os eventos de domínio.
2. **Campos inline** em cada entidade, repetindo o padrão sem abstração compartilhada.
3. **Tabela externa** de "deleções" desnormalizada.

A decisão precisa ser tomada agora porque a EF Configuration e o global query filter dependem dela; mudar depois exige migration + reescrita de queries.

## Decision

**F02 introduz um par padronizado** para soft-delete, usado por `Bloco`, `Unidade` e reutilizado por `Morador` (F03) e `DispositivoPortaria` (F06):

- **Marker interface** `PortaBox.Domain.Abstractions.ISoftDeletable` com propriedades públicas read-only (`bool Ativo { get; }`, `DateTime? InativadoEm { get; }`, `Guid? InativadoPor { get; }`).
- **Classe base abstrata** `PortaBox.Domain.SoftDeleteableAggregateRoot` que estende `AggregateRoot` e implementa `ISoftDeletable`, com métodos protegidos `Inativar(Guid porUserId, DateTime agora)` e `Reativar(Guid porUserId, DateTime agora)` que atualizam os campos e emitem events abstratos sobrescritos pela entidade concreta.
- **Global query filter** aplicado automaticamente em `AppDbContext.OnModelCreating` via reflection (mesmo mecanismo já usado para `ITenantEntity`): para todo tipo que implemente `ISoftDeletable`, aplica `e.Ativo == true`. Consultas que precisam ver inativos usam `.IgnoreQueryFilters()` explicitamente (autorizado em repositórios dedicados de admin/reativação).
- **Unique constraints sensíveis a soft-delete** usam **partial unique index** no Postgres: `UNIQUE (...) WHERE ativo = true`. Registros inativos podem coexistir com reinserções da mesma tripla canônica.

## Alternatives Considered

### Alternative 1: Campos Inline por Entidade

- **Description**: cada entidade declara seus próprios `Ativo`, `InativadoEm`, `InativadoPor`, sem marker nem base class. Query filter configurado explicitamente em cada `IEntityTypeConfiguration`.
- **Pros**: zero abstração; fácil de ler; não acopla entidades a uma hierarquia de classes.
- **Cons**: F03 e F06 vão repetir o mesmo padrão; query filter precisa ser registrado em cada EF Configuration; regras de transição (ex.: não reativar se já existe ativa com mesma tripla) ficam espalhadas; aumenta chance de esquecer o filter em alguma Configuration nova.
- **Why rejected**: economizar abstração hoje para pagar duplicação nas próximas 2 features não vale. A abstração é pequena (~80 linhas) e tem reuso garantido.

### Alternative 2: Tabela Externa de Deleções (`StructuralDeletion`)

- **Description**: entidades não têm `Ativo`; uma tabela separada registra "quais IDs estão atualmente inativos". Consultas de leitura fazem LEFT JOIN/ANTI-JOIN.
- **Pros**: separa concern de "auditoria de deleção" do estado da entidade; útil em sistemas que precisam exibir deleções como timeline.
- **Cons**: toda query precisa conhecer a tabela externa; performance pior (JOIN extra em todo SELECT); EF query filter fica complexo; nenhum benefício real para o caso de F02.
- **Why rejected**: complexidade desproporcional; o use case não demanda separação.

### Alternative 3: Marker interface sem base class

- **Description**: só `ISoftDeletable` com propriedades; entidades implementam manualmente (campos + métodos). Query filter via reflection.
- **Pros**: evita a hierarquia `AggregateRoot → SoftDeleteableAggregateRoot`; mais flexível se alguma entidade soft-deletable não for AggregateRoot (não existe no MVP).
- **Cons**: cada entidade re-implementa os métodos `Inativar`/`Reativar`; eventos de domínio ficam inconsistentes entre entidades; sem um lugar central para imposições (ex.: não pode inativar já inativo).
- **Why rejected**: a classe base adiciona valor real encapsulando regras transversais. O custo da hierarquia é pequeno (herança única em C#, sem multiinheritance conflict).

## Consequences

### Positive

- Padrão único para soft-delete em todo o projeto; novo dev aprende uma vez e reutiliza.
- Global query filter automático via reflection: **impossível esquecer** de filtrar inativos em uma consulta nova (exige `.IgnoreQueryFilters()` consciente).
- Regras transversais (ex.: rejeitar `Inativar()` se já inativo; emitir evento correto) ficam centralizadas na base class.
- Partial unique index no Postgres elimina o conflito canônico ao reinserir após inativação.
- F03 e F06 reaproveitam sem reimplementar.

### Negative

- Hierarquia de classes fica um pouco mais profunda (`Entity → AggregateRoot → SoftDeleteableAggregateRoot → Bloco`). Em casos raros em que a entidade já tem outra necessidade de herança, pode doer.
- Reflection em `OnModelCreating` precisa ser mantida: se futuramente for introduzida uma terceira interface com filter, a mesma infra é reutilizada — precisa ser genérica o suficiente.
- Entidades que **não** devem ter soft-delete (ex.: `TenantAuditEntry`, `DomainEventOutboxEntry`) simplesmente não implementam `ISoftDeletable` — o mecanismo é opt-in; não há risco de contaminar entidades erradas.

### Risks

- **Risco**: global query filter degrada performance em consultas com muitos registros inativos. **Mitigação**: Postgres avalia filters parciais como predicates simples sobre índices — overhead negligível para o volume esperado (10k unidades).
- **Risco**: desenvolvedor esquece de usar `.IgnoreQueryFilters()` em um fluxo que precisa reativar (busca por entidade inativa para reativá-la falha silenciosamente com `NotFound`). **Mitigação**: repositórios expõem métodos explícitos `GetByIdIncludingInactiveAsync(id)` que já fazem `.IgnoreQueryFilters()`. Documentação em xmldoc da base class.
- **Risco**: partial unique index não aceita `NULL` em colunas do índice. **Mitigação**: todas as colunas do índice canônico (`tenant_id`, `bloco_id`, `andar`, `numero`) são NOT NULL por design.

## Implementation Notes

**Interface e base class (abstrações):**

```csharp
// PortaBox.Domain.Abstractions
public interface ISoftDeletable
{
    bool Ativo { get; }
    DateTime? InativadoEm { get; }
    Guid? InativadoPor { get; }
}

// PortaBox.Domain
public abstract class SoftDeleteableAggregateRoot : AggregateRoot, ISoftDeletable
{
    public bool Ativo { get; protected set; } = true;
    public DateTime? InativadoEm { get; protected set; }
    public Guid? InativadoPor { get; protected set; }

    protected Result Inativar(Guid porUserId, DateTime agoraUtc)
    {
        if (!Ativo) return Result.Failure("Já está inativo");
        Ativo = false;
        InativadoEm = agoraUtc;
        InativadoPor = porUserId;
        return Result.Success();
    }

    protected Result Reativar(Guid porUserId, DateTime agoraUtc)
    {
        if (Ativo) return Result.Failure("Já está ativo");
        Ativo = true;
        InativadoEm = null;
        InativadoPor = porUserId;
        return Result.Success();
    }
}
```

**Global query filter em `AppDbContext.OnModelCreating`** (extensão da reflection existente):

```csharp
foreach (var entityType in modelBuilder.Model.GetEntityTypes())
{
    if (typeof(ISoftDeletable).IsAssignableFrom(entityType.ClrType))
    {
        var parameter = Expression.Parameter(entityType.ClrType, "e");
        var property = Expression.Property(parameter, nameof(ISoftDeletable.Ativo));
        var trueConstant = Expression.Constant(true);
        var body = Expression.Equal(property, trueConstant);
        var lambda = Expression.Lambda(body, parameter);
        modelBuilder.Entity(entityType.ClrType).HasQueryFilter(lambda);
    }
}
```

Se a entidade também é `ITenantEntity`, o filtro combinado é aplicado (EF Core compõe automaticamente em `AND`).

**Migration para partial unique index** (exemplo `unidade`):
```sql
CREATE UNIQUE INDEX idx_unidade_canonica_ativa
ON unidade (tenant_id, bloco_id, andar, numero)
WHERE ativo = true;
```
