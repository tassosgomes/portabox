# ADR-009: Endpoint Único `GET /api/v1/condominios/{id}/estrutura` Retornando Árvore Completa

## Status

Accepted

## Date

2026-04-18

## Context

O PRD de F02 estabelece a **árvore hierárquica** (Condomínio → Bloco → Andar → Unidade) como a única visualização no MVP tanto no `apps/sindico` quanto no `apps/backoffice`. O frontend precisa obter os dados para renderizá-la. O desenho do endpoint afeta:

1. **Performance percebida**: um único request que traz tudo de uma vez vs múltiplos requests lazy.
2. **Complexidade do frontend**: estado de expansão/colapso, cache, invalidação pós-mutações.
3. **Custo de invalidação**: pós-mutação (criar unidade, renomear bloco) quais caches precisam ser limpos.
4. **Escalabilidade**: o tenant piloto tem ~200 unidades; o maior condomínio esperado em fases futuras pode chegar a milhares.

Perfil de payload estimado para 300 unidades:
- JSON bruto com `{id, nome, ativo, andar, numero}` por unidade + agregação por bloco = ~30 KB.
- Com compressão gzip (padrão ASP.NET Core Response Compression): ~6 KB.

## Decision

**F02 expõe um único endpoint** para leitura da estrutura:

```
GET /api/v1/condominios/{condominioId}/estrutura?includeInactive={bool}
```

- **Escopo de acesso**:
  - Síndico autenticado do próprio tenant pode chamar (tenant resolution via `TenantResolutionMiddleware`).
  - Operador com role `Operator` pode chamar em nome de qualquer tenant usando `tenantId` explícito na rota; `TenantContext.BeginScope(tenantId)` é aplicado dentro do handler.
- **Resposta**: JSON com a árvore completa já agrupada por bloco e por andar. Blocos ordenados alfabeticamente; dentro do bloco, andares em ordem numérica crescente; dentro do andar, unidades em ordem numérica (com sufixo alfabético) crescente.
- **Query param `includeInactive`** (default `false`): quando `true`, inclui blocos e unidades inativos; responsável pela renderização de inativos é o frontend.
- **Cache**: do lado cliente, TanStack Query com `staleTime` de 30s e invalidação explícita após qualquer mutação de F02 (via `queryClient.invalidateQueries(['estrutura', condominioId])`).
- **Sem paginação** no MVP — a árvore completa cabe em um único response para o volume esperado.

**Endpoints de mutação** seguem CRUD convencional (sem árvore):

```
POST   /api/v1/condominios/{id}/blocos                         (criar bloco)
PATCH  /api/v1/condominios/{id}/blocos/{blocoId}               (renomear bloco)
POST   /api/v1/condominios/{id}/blocos/{blocoId}:inativar      (inativar)
POST   /api/v1/condominios/{id}/blocos/{blocoId}:reativar      (reativar)
POST   /api/v1/condominios/{id}/blocos/{blocoId}/unidades      (criar unidade)
POST   /api/v1/.../unidades/{unidadeId}:inativar               (inativar unidade)
POST   /api/v1/.../unidades/{unidadeId}:reativar               (reativar unidade)
```

Mutações retornam o recurso afetado (não a árvore inteira) + status code apropriado; o frontend invalida o cache da árvore para refetch automático.

## Alternatives Considered

### Alternative 1: Lazy Loading (`/blocos` → `/blocos/{id}/unidades`)

- **Description**: dois endpoints distintos; listar blocos primeiro, só carregar unidades de um bloco quando o usuário expandir.
- **Pros**: escala para cenários com 10k+ unidades sem carregar tudo de uma vez; payload inicial mínimo.
- **Cons**: duplica lógica de cache e invalidação (invalidar unidade exige saber qual bloco); UX de árvore pisca ao expandir blocos (loading spinner por ramo); handler de mutação precisa invalidar dois caches; múltiplos roundtrips no onboarding inicial.
- **Why rejected**: YAGNI para volume real (300 unidades). O overhead de engenharia não compensa; revisitar em Phase 3 se algum condomínio chegar a milhares de unidades.

### Alternative 2: Árvore Por Bloco (`/blocos/{id}/estrutura`)

- **Description**: `GET /blocos` retorna lista de blocos (metadados); `GET /blocos/{id}/estrutura` retorna sub-árvore de um bloco (andares + unidades).
- **Pros**: equilibra performance com granularidade; o frontend pode cache-ar cada sub-árvore independentemente.
- **Cons**: a experiência inicial do usuário exige 1 + N requests (N = blocos do tenant); pós-mutação em unidade invalida só a sub-árvore do bloco, mas exige conhecer o `blocoId` da unidade (factível mas adiciona fio lógico).
- **Why rejected**: ganho marginal sobre (1) para o volume esperado; complexidade acima do mínimo.

### Alternative 3: GraphQL

- **Description**: expõe schema estrutural via GraphQL; cliente escolhe quais campos retornar.
- **Pros**: controle fino sobre payload; natural para árvores.
- **Cons**: F01 não introduziu GraphQL no projeto; adicionaria toolchain inteiro (Hot Chocolate, schema, playgroud); curva de aprendizado; resolvers com N+1 exigem cuidado.
- **Why rejected**: overengineered para o MVP; revisitar só se múltiplas features demandarem query flexível.

## Consequences

### Positive

- **Um único endpoint, uma única query key de cache** (`['estrutura', condominioId]`); invalidação trivial pós-mutação.
- Frontend carrega árvore completa uma vez e renderiza com expand/collapse puro client-side (sem roundtrip).
- Payload ≈ 30 KB é absorvido pelo browser em < 100ms em conexão padrão.
- Handlers de mutação não se preocupam em sincronizar múltiplos caches — apenas retornam o recurso e dão `invalidateQueries` no frontend.
- Cross-tenant no backoffice usa o mesmo contrato; zero divergência entre os dois apps.

### Negative

- Tenant com volume atípico (ex.: 5k unidades) vai ver payload de ~500 KB; pode começar a notar latência de serialização. Mitigação fica para Phase 2 (considerar paginação ou lazy load).
- Toda mutação invalida a árvore inteira no cliente — não há invalidação granular. Para operações em rajada (cadastrar 20 unidades em sequência), o refetch pode ocorrer a cada criação. **Mitigação**: usar `optimistic updates` do TanStack Query para atualizar a árvore local imediatamente, sem refetch.

### Risks

- **Risco**: performance degradada em tenants grandes (1k+ unidades). **Mitigação**: medir em staging; se latência > 500ms consistente, considerar cursor-based pagination por bloco em Phase 2.
- **Risco**: serialização de árvore com muitos inativos sobrecarrega o payload. **Mitigação**: default `includeInactive=false`; quando `true`, o frontend pode pré-avisar "carregando registros arquivados".
- **Risco**: invalidação agressiva causa flicker na UI. **Mitigação**: usar `keepPreviousData` do TanStack Query e optimistic updates.

## Implementation Notes

**Contrato de resposta (JSON):**

```json
{
  "condominioId": "uuid",
  "nomeFantasia": "Condomínio Residencial Alfa",
  "blocos": [
    {
      "id": "uuid",
      "nome": "Bloco A",
      "ativo": true,
      "andares": [
        {
          "andar": 1,
          "unidades": [
            { "id": "uuid", "numero": "101", "ativo": true },
            { "id": "uuid", "numero": "101A", "ativo": true },
            { "id": "uuid", "numero": "102", "ativo": false }
          ]
        },
        {
          "andar": 2,
          "unidades": [ ... ]
        }
      ]
    }
  ],
  "geradoEm": "2026-04-18T12:34:56Z"
}
```

- Query subjacente otimizada: um único `SELECT` com `JOIN` entre `bloco` e `unidade`, ordenado por `bloco.nome, unidade.andar, unidade.numero`. Agrupamento feito em memória no handler (C#).
- Operador autenticado como `Operator` acessa a rota `/admin/condominios/{condominioId}/estrutura` com o `condominioId` do path servindo como tenant explícito (sem query `?tenantId=` adicional); handler valida a role e aplica `ITenantContext.BeginScope(condominioId)` explicitamente (síndico tem scope automático via middleware na rota não-admin).
- Cache do TanStack Query usa chave `['estrutura', condominioId]`. Após mutações, o frontend executa `queryClient.invalidateQueries({ queryKey: ['estrutura', condominioId] })`.
- Backoffice usa a mesma chave + sufixo (`['estrutura-admin', condominioId]`) para isolar cache do app do operador.

**Limitação operacional**: o endpoint não suporta filtro por bloco ou por andar; o frontend filtra client-side. Se no futuro isso virar dor, refatorar para aceitar `?blocoId=` ou `?andar=`.
