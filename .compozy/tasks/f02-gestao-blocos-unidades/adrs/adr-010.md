# ADR-010: Baseline Frontend — TanStack Query (Server State) + React Context (UI State)

## Status

Accepted

## Date

2026-04-18

## Context

F02 é a primeira feature com CRUD cliente-servidor real do projeto. F01 estabeleceu apenas um fluxo transacional de um passo (magic link → definição de senha); o frontend no MVP ainda não tem baseline para gestão contínua de estado de servidor (cache de listas, invalidação pós-mutação, optimistic updates, retry). Essa decisão vai servir de padrão para todas as próximas features frontend (F03 moradores, F04 importação, F08 dispositivos).

A escolha tem quatro candidatos populares em React moderno:

1. **TanStack Query (anteriormente React Query)** — biblioteca dedicada a server state com cache, invalidação, retry, optimistic updates, suspense support, devtools.
2. **SWR (stale-while-revalidate, Vercel)** — similar em espírito, API mais enxuta, menos features avançadas.
3. **Zustand + fetch manual** — store global leve; desenvolvedor implementa cache, invalidação, retry manualmente.
4. **`useState` + `useEffect` + `fetch` cru** — sem abstração dedicada.

Em paralelo, UI state local (aberto/fechado de modais, tema, toasts) é fundamentalmente diferente de server state e pode ser resolvido com React Context + `useState` sem precisar de store global.

## Decision

**F02 adota TanStack Query para server state e React Context para UI state.** Essa combinação é o baseline do frontend PortaBox e deve ser seguida por todas as features subsequentes a menos que um ADR futuro explicitamente altere.

- **Server state**: toda chamada à API PortaBox usa `useQuery` (leituras) ou `useMutation` (escritas). Query keys seguem convenção hierárquica (`['estrutura', condominioId]`, `['moradores', unidadeId]`, ...). Cache com `staleTime` default de 30s; invalidação explícita nos callbacks de mutação (`onSuccess: () => invalidateQueries(...)`).
- **UI state**: Context APIs dedicados para (a) toasts globais (`<ToastProvider>`), (b) tema/tokens (carregados do `portabox-design`), (c) modais singleton que precisam ser disparados de qualquer lugar.
- **Estado local de formulários**: `react-hook-form` (já presente no baseline de F01 para F05). Integração com TanStack Query via `useMutation`.
- **API client**: um pacote dedicado `packages/api-client` encapsula `fetch`, injeta headers (`Authorization`, `Accept-Language`), trata erros via tipo `ApiError` e é consumido pelos hooks de TanStack Query. Gerado **manualmente** no MVP; consideração de geração a partir de OpenAPI fica para Phase 2.

## Alternatives Considered

### Alternative 1: SWR + Context

- **Description**: Vercel SWR cobrindo server state; Context para UI state.
- **Pros**: API minimalista (`const { data } = useSWR(key, fetcher)`); ótimo para leituras; menor curva de aprendizado.
- **Cons**: features avançadas (optimistic updates complexos, mutações agrupadas, infinite queries, suspense) são menos maduras que TanStack Query; devtools menos rico; ecossistema menor; integração com `react-hook-form` menos idiomática.
- **Why rejected**: TanStack Query é mais robusto para o portfólio de features previstas (F04 importação de moradores, F09 dashboard com múltiplas métricas, fluxos de wizard com múltiplos steps mutáveis).

### Alternative 2: Zustand Global Store + Fetch Manual

- **Description**: Zustand para todo estado (server + UI); fetch escrito à mão.
- **Pros**: simplicidade conceitual (uma única store tipada); overhead de bundle mínimo.
- **Cons**: reimplementar cache, invalidação, retry, dedup em cada feature; boilerplate repetitivo; features de UX (stale indicators, background refresh) exigem código custom; difícil fazer optimistic updates corretos.
- **Why rejected**: traz de volta trabalho que TanStack Query resolve há anos. Economizar bundle não vale a dívida de engenharia recorrente.

### Alternative 3: `useState` + `useEffect` + `fetch`

- **Description**: nenhuma biblioteca de dados; React puro.
- **Pros**: zero dependências extra; máximo controle.
- **Cons**: duplicação massiva de código de cache; sem invalidação automática; sem retry; sem optimistic updates; cada componente que consulta dados tem seu próprio useEffect e seu próprio estado — testável mas caro.
- **Why rejected**: não escala para mais que um par de features.

### Alternative 4: Redux Toolkit Query (RTK Query)

- **Description**: ecossistema Redux com RTK Query para server state.
- **Pros**: maduro; integração com Redux DevTools; padrões familiares em equipes com histórico Redux.
- **Cons**: o projeto não usa Redux; introduzir Redux só para RTK Query traz conceitos (store, slices, actions) que não são necessários para o resto da aplicação; bundle maior.
- **Why rejected**: custo conceitual desproporcional sem benefícios sobre TanStack Query.

## Consequences

### Positive

- Server state tratado de forma consistente em todas as features; padrão único reduz tempo de onboarding.
- Cache transparente: listas e árvores não refazem fetch sem necessidade; UX mais rápida.
- Invalidação pós-mutação explícita e rastreável via query keys.
- Optimistic updates disponíveis sem código custom (útil em F02 para refletir criações de unidade sem aguardar refetch).
- Devtools (TanStack Query Devtools) ajudam debugging em dev.
- Integração natural com Suspense (quando/se for adotado em feature futura).

### Negative

- Adiciona ~10 KB gzipped ao bundle. Em bundle total pequeno (portaBox é SPA enxuta), aceitável.
- Curva de aprendizado para quem nunca usou: query keys, staleTime vs cacheTime, invalidação, mutations. Uma hora de onboarding cobre o essencial.
- Abuso de `useQuery` em lugares onde `useState` simples bastaria (modais, toggles) é tentador — disciplina necessária.

### Risks

- **Risco**: query keys inconsistentes entre features (ex.: uma feature usa `['estrutura']` + `condominioId` como segundo arg; outra usa `['estrutura', condominioId]`). **Mitigação**: documentar convenção no README de `apps/sindico`; criar helper `queryKeys.estrutura(condominioId)` para centralizar.
- **Risco**: invalidação agressiva causa refetch em cascata e latência percebida. **Mitigação**: usar optimistic updates onde aplicável; `keepPreviousData` em listas paginadas (futuro).
- **Risco**: `packages/api-client` vira gargalo de build/deploy se mudar muito. **Mitigação**: mantido simples no MVP — um arquivo por recurso (`estrutura.ts`, `moradores.ts`), sem geração automática. Revisitar em Phase 2 quando surgir OpenAPI.

## Implementation Notes

- **Dependências adicionadas** (`apps/sindico` e `apps/backoffice`):
  - `@tanstack/react-query` (core)
  - `@tanstack/react-query-devtools` (dev only)
  - `react-hook-form` (já previsto em F01 para F05)
  - `@hookform/resolvers` + `zod` para validação de formulários (consistente com FluentValidation no backend)
- **Setup inicial** em cada app:
  ```tsx
  // main.tsx
  const queryClient = new QueryClient({
    defaultOptions: { queries: { staleTime: 30_000, retry: 1 } }
  });
  <QueryClientProvider client={queryClient}>
    <App />
    {import.meta.env.DEV && <ReactQueryDevtools />}
  </QueryClientProvider>
  ```
- **Convenção de query keys** (helper centralizado):
  ```ts
  // packages/api-client/src/queryKeys.ts
  export const queryKeys = {
    estrutura: (condominioId: string) => ['estrutura', condominioId] as const,
    estruturaAdmin: (condominioId: string) => ['estrutura-admin', condominioId] as const,
    blocos: (condominioId: string) => ['blocos', condominioId] as const,
  };
  ```
- **API client shape**:
  ```ts
  // packages/api-client/src/http.ts
  export async function apiFetch<T>(path: string, init?: RequestInit): Promise<T> {
    const res = await fetch(`${BASE_URL}${path}`, {
      ...init,
      headers: { 'Content-Type': 'application/json', ...authHeader(), ...init?.headers }
    });
    if (!res.ok) throw await ApiError.fromResponse(res);
    return res.status === 204 ? (undefined as T) : res.json();
  }
  ```
- Provider de toast (UI state) via Context; não usar libs externas (ex.: react-hot-toast) no MVP — componente `<Toast>` do `portabox-design` é a base.
