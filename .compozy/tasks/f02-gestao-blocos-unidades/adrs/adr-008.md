# ADR-008: Auditoria de F02 via Extensão de `TenantAuditEntry.EventKind` + `MetadataJson`

## Status

Accepted

## Date

2026-04-18

## Context

O PRD de F02 exige "100% das operações de escrita com audit trail completo (autor + timestamp + diff onde aplicável)". F01 já introduziu o mecanismo central de auditoria (`TenantAuditEntry`) com schema estável: `Id`, `TenantId`, `EventKind` (enum), `PerformedByUserId`, `OccurredAt`, `Note`, `MetadataJson` (JSONB). Hoje o enum aceita `Created=1, Activated=2, MagicLinkResent=3, Other=4`.

F02 adiciona 7 novos tipos de operação auditável (bloco criado/renomeado/inativado/reativado; unidade criada/inativada/reativada). A decisão é onde materializar essa auditoria:

1. **Estender o enum e reutilizar a tabela existente** (adicionar valores 5–11 a `EventKind`, encodar detalhes em `MetadataJson`).
2. **Criar tabela própria** (`EstruturaAuditEntry`) com schema tipado para o domínio estrutural.
3. **Eliminar auditoria dedicada** e confiar nos domain events persistidos no outbox.

A escolha afeta o tipo de UI que o backoffice pode oferecer, a facilidade de consulta por tenant/ator e a complexidade da migration.

## Decision

**F02 estende `TenantAuditEntry.EventKind`** com os valores:

| Valor | Nome do kind |
|---:|---|
| 5 | `BlocoCriado` |
| 6 | `BlocoRenomeado` |
| 7 | `BlocoInativado` |
| 8 | `BlocoReativado` |
| 9 | `UnidadeCriada` |
| 10 | `UnidadeInativada` |
| 11 | `UnidadeReativada` |

Cada operação materializa uma linha em `TenantAuditEntry` com:

- `PerformedByUserId` = síndico que executou a ação.
- `OccurredAt` = timestamp do commit (preenchido pelo handler).
- `Note` = curta descrição humana ("Bloco 'Torre Alfa' renomeado").
- `MetadataJson` = payload estruturado com diff/contexto, esquema padronizado por kind:
  - `BlocoCriado`: `{ "blocoId": "...", "nome": "Torre Alfa" }`.
  - `BlocoRenomeado`: `{ "blocoId": "...", "nomeAntes": "Bloco A", "nomeDepois": "Torre Alfa" }`.
  - `BlocoInativado` / `BlocoReativado`: `{ "blocoId": "...", "nome": "Torre Alfa" }`.
  - `UnidadeCriada`: `{ "unidadeId": "...", "blocoId": "...", "andar": 2, "numero": "201" }`.
  - `UnidadeInativada` / `UnidadeReativada`: `{ "unidadeId": "...", "blocoId": "...", "andar": 2, "numero": "201" }`.

Os handlers de F02 são responsáveis por criar a entry na mesma transação da mutação de estado (padrão já estabelecido em F01 pelo `CreateCondominioCommandHandler`).

## Alternatives Considered

### Alternative 1: Tabela Dedicada `EstruturaAuditEntry`

- **Description**: nova tabela com colunas tipadas (ex.: `BlocoId`, `AntesNome`, `DepoisNome`, `UnidadeId`, `Andar`, `Numero`, `Kind` enum curto).
- **Pros**: schema explícito; queries tipadas mais ergonômicas ("todos os blocos renomeados no último mês" sem parsing de JSONB); índices específicos.
- **Cons**: duas tabelas de auditoria para o mesmo tenant; o backoffice passa a consultar ambas para exibir "timeline de operações no tenant"; migrations crescem; manutenção do modelo em evolução exige mudar schema em vez de apenas adicionar enum value.
- **Why rejected**: fragmenta a história do tenant sem ganho real. JSONB do Postgres é indexável (GIN) e queries são suficientemente ergonômicas para o volume esperado.

### Alternative 2: Sem Auditoria Dedicada (Confiar no Outbox)

- **Description**: domain events (`bloco.criado.v1`, `bloco.renomeado.v1`, ...) já persistem no `DomainEventOutbox`. Backoffice consome outbox para exibir histórico; nenhuma escrita em `TenantAuditEntry` para F02.
- **Pros**: elimina duplicação (evento + audit entry carregam a mesma info); menos escrita por mutação.
- **Cons**: outbox é tabela de **entrega** de eventos, não de **consulta** histórica; entries podem ser purgadas após publicação em fases futuras; consultar "quem mexeu neste bloco?" exige parsing de payload de eventos distintos; o schema de evento pode evoluir e quebrar queries históricas.
- **Why rejected**: mistura responsabilidades (entrega de eventos vs auditoria) e perde o contrato claro que F01 estabeleceu (`TenantAuditEntry` é a fonte única de trilha de auditoria).

## Consequences

### Positive

- Auditoria concentrada em uma única tabela do tenant; backoffice tem um único lugar para consultar timeline.
- Adicionar valores ao enum é uma migration simples (sem novo schema); evolução futura (auditar F03, F06) é barata.
- `MetadataJson` aceita evolução de payload sem migration; um novo campo no diff é só escrita — leitores antigos ignoram campos desconhecidos.
- Consulta de "tudo que aconteceu com esse tenant no último mês" responde em uma query contra uma única tabela + índices.

### Negative

- `MetadataJson` impõe disciplina: cada handler deve seguir o schema acordado por kind, senão queries por diff ficam inconsistentes. Necessário documentar contratos em xmldoc de um helper `StructuralAuditMetadata.For(...)`.
- UI de auditoria precisa lógica específica por kind para renderizar o diff legível — não é um render genérico.

### Risks

- **Risco**: handlers de F02 esquecem de criar a audit entry. **Mitigação**: padrão codificado em helper (`IAuditService.RecordAsync(kind, metadata, ...)` ou métodos dedicados); testes de integração validam a escrita.
- **Risco**: tamanho da tabela cresce linearmente com operações do tenant. Para um condomínio ativo com 200 unidades e movimento regular, expectativa é ~ poucos milhares de linhas/ano. **Mitigação**: índice por `(tenant_id, occurred_at desc)` já existe; particionamento por mês é opção futura se volume exigir.
- **Risco**: payload JSONB inconsistente (um handler escreve `blocoId`, outro `bloco_id`). **Mitigação**: helper centralizado + testes de schema em unit tests.

## Implementation Notes

- A migration adiciona valores ao enum no C#; não há mudança na coluna `event_kind` no Postgres (já é `smallint`). O enum no EF mantém o `HasConversion<short>()` existente.
- Handlers recebem `IAuditService` por DI; o serviço encapsula a criação da entry (`auditService.RecordStructuralAsync(kind, tenantId, performedBy, metadata, ct)`).
- Transacionalidade: `IAuditService.RecordAsync` apenas adiciona ao `DbContext`; o commit é feito junto com a mutação da entidade.
- Consulta padrão no backoffice: `GET /api/v1/admin/condominios/{condominioId}/audit?kind=BlocoRenomeado&since=...` (endpoint novo; Phase 2 — hoje apenas documentado como open question no `api-contract.md`).
