# ADR-006: F02 Reutiliza o Módulo `PortaBox.Modules.Gestao`

## Status

Accepted

## Date

2026-04-18

## Context

F02 introduz duas entidades centrais do domínio D01 — `Bloco` e `Unidade` — que serão consumidas por F03 (moradores), F04 (importação), F07 (API interna) e D04 (fuzzy matching). A pergunta arquitetural é se essas entidades devem viver no módulo existente `PortaBox.Modules.Gestao` (onde F01 já colocou `Condominio`, `Sindico`, `OptInRecord`, `MagicLink`) ou em um módulo novo (`PortaBox.Modules.Estrutura` / `PortaBox.Modules.Blocos`).

A decisão afeta:

1. **Custos de estrutura**: novo módulo exige `.csproj`, `DependencyInjection.cs`, suite de arch tests clonada, referências entre projetos cuidadosamente configuradas.
2. **Domain boundaries**: um único bounded context de "D01 — Gestão do Condomínio" comporta `Condominio` + `Bloco` + `Unidade` + `Morador` sem conflito conceitual; fragmentar em sub-módulos é uma decisão sobre granularidade de deploy/rebuild, não sobre domínio.
3. **DbContext e migrations**: módulo único → um `AppDbContext` por módulo, uma sequência de migrations linear. Módulos separados → DbContext compartilhado ou múltiplos com sincronização.
4. **Velocidade de entrega**: F02 é parte da Fase 1 do MVP com alto acoplamento temporal a F03/F04; fragmentar atrasaria.

## Decision

**F02 reutiliza integralmente o módulo `PortaBox.Modules.Gestao`.** Novas entidades (`Bloco`, `Unidade`), repositórios (`IBlocoRepository`, `IUnidadeRepository`), commands, queries, handlers, validators, EF configurations e eventos de domínio são adicionados dentro do mesmo csproj, no mesmo `AppDbContext`, registrados em `PortaBox.Modules.Gestao.DependencyInjection.AddPortaBoxModuleGestao`.

Nenhum novo `PortaBox.Modules.*` é criado no escopo de F02. A organização interna segue a convenção já estabelecida em F01 (pastas `Domain/`, `Application/`, `Infrastructure/`), com sub-pastas por agregado (`Domain/Blocos/`, `Domain/Unidades/`) para manter navegabilidade.

## Alternatives Considered

### Alternative 1: Novo módulo `PortaBox.Modules.Estrutura`

- **Description**: criar `src/PortaBox.Modules.Estrutura/` com csproj, DI, arch tests e DbContext próprios (ou compartilhando `AppDbContext` em Infrastructure).
- **Pros**: separação física alinhada a uma eventual extração futura para microsserviço; menor acoplamento de build entre onboarding e estrutura.
- **Cons**: overhead imediato (3 arquivos novos de projeto + arch tests duplicados); duplica a sequência de migrations se DbContext for separado; `Bloco` referencia `Condominio` como FK — hoje no mesmo módulo, teria que atravessar fronteira; ganho de modularização prematuro.
- **Why rejected**: YAGNI. Não há sinal de que D01 vai ser fragmentado nem extraído em fase próxima; o custo é imediato e a contrapartida é hipotética.

### Alternative 2: Sub-namespace interno dentro de `PortaBox.Modules.Gestao`

- **Description**: mesmo csproj, mas namespaces explicitamente separados (`PortaBox.Modules.Gestao.Tenants.*` para F01, `PortaBox.Modules.Gestao.Structural.*` para F02).
- **Pros**: organiza código sem fragmentar build; facilita futura extração.
- **Cons**: convenção cosmética que depende de disciplina; arch tests não conseguem impor boundary entre sub-namespaces do mesmo csproj sem custom rules; dificulta casos legítimos em que F01 precisa referenciar F02 (ex.: handler de ativação do tenant conferindo se há ≥1 bloco ativo no futuro).
- **Why rejected**: complica sem ganho mensurável. A convenção de pastas por agregado (`Domain/Blocos/`, `Domain/Unidades/`) já dá a navegabilidade desejada sem criar fronteira artificial.

## Consequences

### Positive

- Zero overhead de projeto novo; F02 consome toda a infra já implementada em F01 (AppDbContext, interceptor de outbox, migrations pipeline, arch tests).
- `Bloco` → `Condominio` e `Unidade` → `Bloco` são FKs dentro do mesmo módulo, sem atravessar boundaries.
- Um único DI para o módulo D01 (`AddPortaBoxModuleGestao`) — consumidores (Program.cs) não precisam mudar.
- Migrations geradas no mesmo pipeline; release notes do piloto continuam concentradas em um módulo.

### Negative

- `PortaBox.Modules.Gestao` cresce além de um único contexto de feature — ficará maior ao longo das próximas features de D01. Disciplina de sub-pastas e testes por agregado precisa ser mantida para evitar que o csproj vire "um saco" de código desconexo.
- Se no futuro houver decisão de split (ex.: por razão de release cycle ou performance de build), a refatoração será maior que se tivéssemos começado separado. Aceitamos esse custo futuro hipotético contra o benefício presente certo.

### Risks

- **Risco**: acoplamento excessivo entre lógica de `Condominio` e lógica de `Bloco/Unidade` (ex.: handler de ativação do tenant acaba conhecendo estrutura). **Mitigação**: manter cada agregado em sua sub-pasta, com commands/queries próprios; comunicação entre agregados só via eventos de domínio.
- **Risco**: volume de handlers em `DependencyInjection.cs` cresce muito. **Mitigação**: considerar futuro refator para DI por extensão (`services.AddBlocoHandlers()` interno ao módulo) se ficar visualmente desconfortável — não no MVP.

## Implementation Notes

- Organização de pastas dentro de `PortaBox.Modules.Gestao`:
  ```
  Domain/
    Tenants/          (Condominio, Sindico, OptIn — F01 existente)
    Blocos/           (Bloco, eventos)
    Unidades/         (Unidade, eventos)
  Application/
    Tenants/          (commands, queries, handlers de F01)
    Blocos/
    Unidades/
    Common/           (PagedResult, filtros reutilizados)
  Infrastructure/
    EfConfigurations/ (CondominioConfiguration, BlocoConfiguration, UnidadeConfiguration)
    Repositories/     (CondominioRepository, BlocoRepository, UnidadeRepository)
  ```
- Arch tests (`tests/PortaBox.Modules.Gestao.UnitTests/ArchitectureTests.cs`) permanecem inalterados — as regras (módulo não referencia Infrastructure/Api) continuam válidas para o código novo.
- Novo csproj de testes não é necessário; adicionar casos ao existente.
