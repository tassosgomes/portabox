# ADR-010: Frontend Adere ao Design System `portabox-design` (Tokens, Tipografia, Componentes e Copy pt-BR)

## Status

Accepted

## Date

2026-04-17

## Context

F01 entrega duas superfícies web React:

1. **`apps/backoffice`** — SPA do operador interno (wizard 3 etapas, lista, detalhes do tenant).
2. **`apps/sindico`** — esqueleto mínimo para validar o fluxo do magic link (tela de definição de senha, login, home placeholder).

O produto tem um design system interno versionado em `.claude/skills/portabox-design/` (skill user-invocable `portabox-design`), com tokens, tipografia, imagery, kits de componentes (`porter-app`, `resident-message`, `admin-dashboard`) e regras firmes de copy pt-BR. A decisão é se adotamos esse sistema como fonte única de verdade desde o primeiro pixel do projeto ou se começamos com uma biblioteca de terceiros (shadcn/ui, Material, Fluent) e convergimos depois.

## Decision

O frontend deste (e dos próximos) PRDs do projeto **adere ao design system `portabox-design`** como fonte única de verdade para o visual e a voz do produto:

- **Tokens obrigatórios**: importar `colors_and_type.css` do skill no Vite build e expor os tokens como variáveis CSS em `:root`. Toda cor, tipografia, espaçamento, radius e sombra usada no código consome os tokens (`var(--pb-navy-700)`, `var(--r-lg)`, `var(--sp-4)` etc.). **Nenhum valor de cor ou tipografia é hardcoded**.
- **Tipografia**: Plus Jakarta Sans (display/UI), Inter (body), JetBrains Mono (PIN/IDs). Carregadas do Google Fonts conforme `colors_and_type.css`.
- **Paleta**: Navy (`#0B2B47`, `#1E3A8A`) + laranja (`#F97316`, `#EA580C`) como únicas cores de marca; ícones em `currentColor`; sem gradientes; sem acentos secundários.
- **Ícones**: [Lucide](https://lucide.dev) via `lucide-react` — stroke 1.5–2px, sempre com label em navegação e botões.
- **Componentes**: as telas do F01 (formulários de wizard, botões, inputs, badges, cards, modais) replicam o visual do kit `ui_kits/admin-dashboard/` já que é a outra superfície desktop web do produto. Antes de implementar qualquer componente novo, o desenvolvedor inspeciona `preview/component-*.html` do skill para referência.
- **Copy**: pt-BR, segunda pessoa "você", sentence case, tom acolhedor-utilitário. Vocabulário-padrão: `encomenda`, `morador`, `portaria`, `síndico`, `retirada`, `Token de Retirada`. `📦` **apenas** em mensagens ao morador (WhatsApp/SMS) — nunca na UI interna.
- **Imagery**: quando houver, fotos reais dos `assets/` do skill (condomínios brasileiros, luz natural). Nada de ilustrações de pacotes, 3D renders ou stock abstrato.
- **Motion**: 200ms padrão, ease `cubic-bezier(0.2, 0.7, 0.2, 1)`, sem bounce/spring. Fades + translate 4–8px.
- **Focus/hover/press**: halo laranja `rgba(249, 115, 22, 0.35)` em focus; hover muda para `--pb-orange-700`; press `scale(0.98)`.
- **Acessibilidade**: contraste AA mínimo; hit targets ≥ 44×44 (mobile futuro); foco sempre visível.
- **Logo**: `assets/logo-portabox.png` para lockup completo, `assets/logo-portabox-mark.svg` para favicon e chrome pequeno. Nunca reconstruir ou recolorir.
- **Fluxo de uso da skill**: antes de qualquer implementação frontend, o desenvolvedor (humano ou agente) invoca a skill `portabox-design` para carregar o contexto. Para protótipos/HTMLs descartáveis de exploração, seguir orientação da skill (copy assets out, static HTML). Para código de produção, os tokens são consumidos via CSS vars.

## Alternatives Considered

### Alternative 1: Começar com biblioteca de terceiros (shadcn/ui, MUI, Chakra)

- **Description**: adotar um component library pronto e customizar tema depois.
- **Pros**: velocidade inicial; componentes acessíveis out-of-the-box.
- **Cons**: rework ao migrar para o design system real; risco de cair em visuais genéricos ("AI aesthetic") que o próprio skill `portabox-design` orienta a evitar; dependência extra no bundle.
- **Why rejected**: o design system já existe com suficiente detalhe (tokens, kits, regras firmes); começar com ele desde o dia 1 evita dívida de design.

### Alternative 2: Criar um design system inline ad-hoc

- **Description**: tokens e componentes criados de forma oportunista dentro do repo.
- **Pros**: zero dependência externa.
- **Cons**: ignora ativo existente (`portabox-design`); cada desenvolvedor tomaria microdecisões divergentes; dificilmente chegaria ao nível de cuidado do skill.
- **Why rejected**: desperdício do trabalho de design já feito.

### Alternative 3: Copiar os tokens do skill para dentro do repo (fork)

- **Description**: replicar `colors_and_type.css` dentro de `apps/shared/design-tokens/` e manter sincronia manual.
- **Pros**: remove dependência do diretório do skill em build.
- **Cons**: duplicação leva a drift entre a skill e o código de produção; atualizações no design system precisam ser copiadas duas vezes.
- **Why rejected**: a skill é o upstream; manter uma única fonte evita drift. A skill fica versionada no repo (`.claude/skills/portabox-design/`) e o build Vite aponta para ela.

## Consequences

### Positive

- Visual consistente entre backoffice, painel do síndico e qualquer nova superfície web desde a primeira linha de código.
- Copy pt-BR alinhada; vocabulário do produto reforçado.
- Onboarding de novos devs mais rápido (um único documento de referência).
- Agentes de código (Cursor, Claude Code) têm contexto explícito via skill.

### Negative

- Exige disciplina de consumir tokens em vez de valores hardcoded (fácil de violar em pressa).
- Qualquer evolução do design system precisa ser feita na skill e propagada (não é um problema — é o desenhado).

### Risks

- **Risco:** desenvolvedor desconhece a skill e implementa UI genérica. **Mitigação:** referência explícita a este ADR e ao skill no README do `apps/backoffice`; linter que alerta ao detectar `color:` hardcoded fora de tokens; checklist de PR exige invocação da skill na task de frontend.
- **Risco:** Plus Jakarta Sans / Inter não são as fontes "oficiais" finais (flag no README do skill). **Mitigação:** consumir via token de família (`var(--font-display)`); troca futura é um único edit no CSS.
- **Risco:** o kit `admin-dashboard` é explicitamente marcado como "first-pass" no skill. **Mitigação:** usar como referência visual mas não copiar componentes 1:1; tratar divergências como feedback para o skill.

## Implementation Notes

- `apps/backoffice/index.html` e `apps/sindico/index.html` importam as fontes via `<link>` para Google Fonts (ou via `@import` em CSS global).
- `apps/backoffice/src/styles/tokens.css` reexporta o `colors_and_type.css` da skill ou importa-o diretamente via path relativo do monorepo (`@import url('../../../.claude/skills/portabox-design/colors_and_type.css')`).
- `lucide-react` adicionado no `package.json` dos dois apps.
- Componentes comuns (Button, Input, Card, Badge, Modal, StepIndicator) ficam em `packages/ui/` (monorepo shared), seguindo os exemplos de `preview/component-*.html` da skill.
- Naming em inglês no código; labels e copy em pt-BR nos strings (react-i18next não é necessário no MVP — strings literais em pt-BR bastam).
- Antes de fazer merge de uma tela nova, comparar visualmente com o kit `admin-dashboard` do skill; divergências intencionais precisam de justificativa no PR.
