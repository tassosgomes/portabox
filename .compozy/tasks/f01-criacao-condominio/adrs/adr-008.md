# ADR-008: `IEmailSender` sobre SMTP Genérico no MVP (Provedor Comercial Adiado)

## Status

Accepted

## Date

2026-04-17

## Context

F01 dispara e-mail transacional com o magic link ao primeiro síndico (CF3 do PRD). Decisão a tomar: qual provedor comercial adotar já no MVP? Alternativas evidentes: AWS SES, SendGrid, Resend ou SMTP genérico com abstração para trocar depois.

## Decision

- **Interface `IEmailSender`** na camada de aplicação com contrato minimalista: `Task SendAsync(EmailMessage message, CancellationToken ct)` onde `EmailMessage` contém `To`, `Subject`, `HtmlBody`, `TextBody` e `Tags` (opcionais, usados para observabilidade).
- **Adapter único no MVP**: `SmtpEmailSender` que usa `MailKit` (NuGet `MailKit`) e lê configuração de `EmailOptions` (`Host`, `Port`, `Username`, `Password`, `FromAddress`, `UseStartTls`).
- **Ambiente dev/local**: container [MailHog](https://github.com/mailhog/MailHog) ou [MailPit](https://github.com/axllent/mailpit); nenhum e-mail real sai durante desenvolvimento.
- **Ambiente prod**: SMTP relay (a provedora de SMTP real é uma **Open Question** a ser resolvida antes do primeiro go-live; opções candidatas: SES SMTP endpoint, Mailgun SMTP, Resend SMTP, Postmark SMTP).
- **Abstração preserva migração**: quando a decisão de provedor comercial for tomada, podemos optar por (a) continuar com SMTP desse provedor (sem refactor) ou (b) criar adapter novo sobre a API nativa (ex.: `SesEmailSender`) e trocar via DI.
- **Templates de e-mail**: HTML simples em arquivos embutidos (`Resources/EmailTemplates/*.cshtml` ou strings estáticas) renderizados com Scriban/Razor; suficientes para o MVP (1 template: "Defina sua senha").
- **Retry e DLQ**: envio é tentado no handler em síncrono; em falha, cai em retry exponencial limitado (Polly — skill `dotnet-dependency-config`) e, persistindo a falha, grava na tabela `email_outbox` para retry posterior por worker em background. Padrão coerente com ADR-009.

## Alternatives Considered

### Alternative 1: AWS SES desde o MVP

- **Description**: escolher SES agora, usar SDK nativo AWS.
- **Pros**: provedor maduro e barato.
- **Cons**: obriga ter conta AWS configurada antes do desenvolvimento local; SES em sandbox exige verificação de identidade; acopla ao ecossistema AWS.
- **Why rejected para MVP**: SMTP genérico permite desenvolver sem a dependência externa e trocar depois.

### Alternative 2: SendGrid / Resend / Postmark

- **Description**: provedor SaaS dedicado com API própria.
- **Pros**: templates em UI, boa deliverability, analytics.
- **Cons**: vendor lock-in antes de haver volume que justifique; decisão comercial prematura.
- **Why rejected para MVP**: o mesmo racional da alternativa 1.

## Consequences

### Positive

- Implementação começa sem dependência externa; dev/test usam MailHog.
- Qualquer provedor que ofereça SMTP relay pode ser plugado sem refactor.
- Abstração permite migrar para API nativa quando/se necessário.

### Negative

- Funcionalidades avançadas (webhooks de bounce/complaint, templating remoto) não estão disponíveis no SMTP puro — ficam pendentes para quando o provedor for escolhido.

### Risks

- **Risco:** deliverability ruim se usar SMTP relay gratuito/sem reputação. **Mitigação:** escolher provedor com boa reputação antes de produção; adicionar SPF/DKIM/DMARC no domínio corporativo.
- **Risco:** e-mails de magic link caem em spam. **Mitigação:** texto claro, From name identificável, link com HTTPS, configurar `Reply-To` adequado.

## Implementation Notes

- NuGet: `MailKit` para SMTP; `Polly` para retry (já na baseline).
- Em ambiente de teste: `FakeEmailSender` que registra em memória para asserts de integração.
- Endpoint `GET /api/v1/admin/emails` (apenas em dev) expõe últimas N mensagens para depuração.
