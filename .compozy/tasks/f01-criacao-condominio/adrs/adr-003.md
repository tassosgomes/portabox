# ADR-003: CNPJ Obrigatório como Identificador Canônico do Condomínio

## Status

Accepted

## Date

2026-04-17

## Context

A plataforma é multi-tenant desde o dia 1 (RN-01). Precisamos de uma chave canônica para:

- Evitar cadastro duplicado do mesmo condomínio.
- Suportar, em fases futuras, relação com administradoras (um CNPJ de condomínio pode estar associado a um CNPJ de administradora).
- Permitir conciliação com registros externos (fiscais, cartório, bancos) quando necessário.

Condomínios residenciais no Brasil, quando constituídos formalmente, possuem CNPJ próprio (Código 8112-5/00 — "Administração de condomínios prediais"). Condomínios pequenos ou informais podem operar sem CNPJ, mas esse cenário é marginal no público-alvo da plataforma (100+ unidades).

## Decision

F01 exige o **CNPJ do condomínio como campo obrigatório** no wizard, com:

- Validação de formato e dígito verificador no momento do preenchimento.
- Bloqueio automático de cadastro duplicado (não permite criar dois tenants com o mesmo CNPJ).
- O CNPJ passa a ser o identificador canônico externo do tenant (o identificador interno continua sendo um ID opaco gerado pelo sistema).

## Alternatives Considered

### Alternative 1: CNPJ opcional + identificador canônico apenas interno

- **Description**: CNPJ é campo opcional; chave canônica é ID interno; detecção de duplicatas feita via alerta por similaridade de nome + endereço, sem bloqueio.
- **Pros**: inclusão de condomínios pequenos ou informais sem CNPJ formalizado.
- **Cons**: alerta por similaridade é falível (diferentes escritas, endereços imprecisos); risco real de criar dois tenants para o mesmo condomínio; dificulta futura integração com administradoras (que geralmente relacionam clientes por CNPJ).
- **Why rejected**: público-alvo declarado (100+ unidades) sempre tem CNPJ; o custo de inclusão dos casos marginais não compensa a insegurança de identificação.

### Alternative 2: Apenas nome + endereço, sem CNPJ

- **Description**: cadastro mínimo sem CNPJ nem dados fiscais.
- **Pros**: zero fricção para casos informais.
- **Cons**: praticamente impossível deduplicar; inviabiliza integrações futuras com administradoras e ERPs; fere a arquitetura multi-tenant sólida.
- **Why rejected**: inadequado para o público-alvo.

## Consequences

### Positive

- Deduplicação confiável desde o dia 1.
- Preparação limpa para Fase 3 (administradoras operando múltiplos tenants).
- Registro canônico auditável por inspeções fiscais ou jurídicas.

### Negative

- Condomínios sem CNPJ formalizado não conseguem usar a plataforma. Mitigação: documentar como pré-requisito de onboarding; administradora pode orientar constituição se cliente ainda não tem.

### Risks

- **Risco:** CNPJ digitado errado pelo operador passa pela validação de dígito (é um CNPJ válido, mas não é o desse condomínio). **Mitigação:** operador confirma com o síndico/administradora antes de salvar; confirmação dupla no último passo do wizard.
- **Risco:** tentativa de cadastrar CNPJ já existente gera erro ambíguo (pode ser o mesmo condomínio ou erro de digitação). **Mitigação:** mensagem clara mostrando o tenant existente (nome, data de criação) para decisão do operador.

## Implementation Notes

A validação de dígito verificador é padrão (algoritmo público). Normalização deve armazenar CNPJ sem máscara (14 dígitos). Eventual integração com API da Receita Federal para pré-preenchimento de razão social é desejável, porém fora do escopo do MVP (Open Question).
