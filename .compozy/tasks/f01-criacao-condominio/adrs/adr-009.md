# ADR-009: Eventos de Domínio In-process no MVP com Outbox Pattern para Publicação Futura

## Status

Accepted

## Date

2026-04-17

## Context

O domain doc de D01 declara eventos produzidos pelo domínio: `condominio.cadastrado`, `morador.cadastrado`, `morador.atualizado`, `morador.inativado`, `dispositivo-portaria.vinculado`, `dispositivo-portaria.revogado`. O vision.md fixa que a comunicação entre o serviço .NET e o Mastra (D04) é assíncrona (evento/fila), e a skill `dotnet-dependency-config` indica RabbitMQ com `Rmq.CloudEvents` como baseline futuro.

No MVP, porém, apenas F01 e features subsequentes de D01 serão entregues. O D04 (Mastra) ainda não existe. Implementar RabbitMQ + schemas CloudEvents + subscribers agora, sem consumer real, é overhead sem benefício imediato. Por outro lado, ignorar o tema hoje gera refactor pesado quando o Mastra entrar.

## Decision

- **Dispatcher in-process no MVP**: eventos são publicados via um `IDomainEventDispatcher` injetável; no MVP ele invoca handlers registrados no mesmo processo (resolvidos via DI). Útil para hooks internos (ex.: criação de `Condominio` dispara handler que agenda o envio do magic link do síndico).
- **Outbox Pattern desde já** para eventos que cruzam fronteira de serviço: ao salvar um `AggregateRoot` com eventos pendentes, o repositório grava **na mesma transação** uma linha em `domain_event_outbox` (id, aggregate_id, event_type, payload JSON, created_at, published_at nullable). Garante consistência atômica entre estado + evento registrado.
- **Worker de publicação**: no MVP, worker lê outbox a cada N segundos e apenas atualiza `published_at = now()` (não publica em lugar nenhum real). Isso exercita o pipeline e popula dados auditáveis sem exigir broker real.
- **Evolução para RabbitMQ**: quando D04/Mastra entrar, o worker passa a publicar em exchange RabbitMQ usando envelope [CloudEvents v1.0](https://cloudevents.io/) (conforme `Rmq.CloudEvents` da skill). Zero mudança no modelo de domínio ou repositórios; apenas a implementação do worker troca.
- **In-process dispatcher vs Outbox**: uso é complementar. Handlers puramente internos (ex.: enviar e-mail via `IEmailSender`) continuam in-process. Eventos que outros domínios/serviços vão consumir passam pela outbox.

## Alternatives Considered

### Alternative 1: RabbitMQ desde o dia 1

- **Description**: configurar RabbitMQ + Rmq.CloudEvents desde o MVP; publicar eventos em exchange real; subscribers dummies ou ausentes.
- **Pros**: pipeline completa exercitada cedo; quando Mastra entrar é só plugar consumer.
- **Cons**: infra adicional (broker) em dev/local/prod para pouco ou nenhum benefício no piloto; custo operacional e de deploy antecipado.
- **Why rejected**: overkill para o estado atual; outbox pattern já garante o principal risco (atomicidade evento+estado).

### Alternative 2: Sem eventos de domínio; leitura direta entre domínios via API interna

- **Description**: domínios se comunicam exclusivamente por chamadas síncronas a APIs internas.
- **Pros**: menos abstrações.
- **Cons**: contradiz a arquitetura declarada (vision.md: "comunicação assíncrona"); introduz acoplamento temporal entre domínios; dificulta futuros workflows agênticos do Mastra.
- **Why rejected**: diverge da visão de produto.

### Alternative 3: Outbox + polling dentro do mesmo monólito (sem worker separado)

- **Description**: outbox existe, mas o próprio request que gerou o evento já publica antes de retornar.
- **Pros**: simplicidade extra.
- **Cons**: fere o propósito do outbox (desacoplar publicação de transação); em caso de crash entre commit e publish, eventos se perdem.
- **Why rejected**: perde os benefícios centrais do padrão.

## Consequences

### Positive

- Consistência atômica (estado + evento) desde o dia 1, com custo de infra zero.
- Transição para RabbitMQ é apenas trocar a implementação do worker de publicação.
- Auditoria natural via outbox (todos os eventos ficam persistidos).

### Negative

- Outbox não publica em lugar real no MVP — é "código morto útil". Risco de regressão silenciosa (worker quebra mas ninguém percebe). Mitigação: monitoramento da fila de eventos não publicados (idade máxima + contagem).
- Handlers in-process compartilham transação/contexto do request, exigindo disciplina para não virarem "god-handlers".

### Risks

- **Risco:** schema drift dos eventos conforme features evoluem antes da integração com Mastra. **Mitigação:** `event_type` versionado (`condominio.cadastrado.v1`) desde o primeiro evento emitido; changelog documentado em `domains/<domínio>/events.md` (a criar conforme necessário).
- **Risco:** volume da tabela outbox crescendo indefinidamente. **Mitigação:** job de housekeeping move `published_at IS NOT NULL` para tabela fria após X dias (fora do escopo do MVP; Open Question).

## Implementation Notes

- `AggregateRoot` base classe com método `AddDomainEvent(IDomainEvent e)` e propriedade `DomainEvents` (IReadOnlyList).
- `SaveChangesInterceptor` do EF Core: antes do commit, move `DomainEvents` para inserts de `domain_event_outbox` e limpa a coleção.
- Dispatcher in-process escuta `AfterSave` e executa handlers locais (ex.: orquestração intra-serviço).
- Worker de publicação é um `BackgroundService` do ASP.NET Core; pode ser desabilitado via flag de configuração (`DomainEvents:Publisher:Enabled`).
