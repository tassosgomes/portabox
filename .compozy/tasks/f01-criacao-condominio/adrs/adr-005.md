# ADR-005: Backoffice como SPA React Separado; Autenticação via ASP.NET Core Identity com Roles `Operator` e `Sindico`

## Status

Accepted

## Date

2026-04-17

## Context

O PRD de F01 define duas personas do painel web:
- **Operador interno da plataforma**: usa o backoffice para criar e ativar tenants (fluxo do F01).
- **Síndico do condomínio**: usa o painel web para gerenciar blocos/unidades/moradores (fluxos de F02, F03, F04 etc.).

As duas personas não têm sobreposição funcional, têm públicos completamente distintos e requisitos de segurança diferentes (operador tem acesso cross-tenant; síndico é amarrado a um único tenant). Há também uma decisão adjacente de autenticação, já que F01 precisa criar a credencial inicial do primeiro síndico e dar ao operador acesso ao backoffice.

## Decision

1. **Duas SPAs React independentes** em repositório monorepo (`apps/backoffice` e `apps/sindico`), buildadas e servidas em hosts/subdomínios distintos. O bundle do backoffice nunca chega ao browser do síndico ou do morador.
2. **ASP.NET Core Identity** como solução de autenticação para ambas as personas, compartilhando uma única tabela `AspNetUsers` e discriminando acesso por **role** (`Operator`, `Sindico`). Políticas de autorização (`[Authorize(Roles = "Operator")]`) governam os endpoints do backoffice; `[Authorize(Roles = "Sindico")]` governa os do painel do síndico.
3. **Usuário `Operator` é provisionado via seed/migração** (piloto) ou via um endpoint interno restrito acessível apenas a um `Operator` já existente; nunca via auto-registro público.
4. **Usuário `Sindico` é criado pelo fluxo F01**: operador fornece nome/e-mail/celular no wizard, o sistema cria a conta em estado `PendingPasswordSetup` (sem `PasswordHash`), e dispara o magic link (ver ADR-006).
5. **Autenticação segue padrão cookie-based** dentro de cada SPA (Cookie Authentication Middleware do ASP.NET Core Identity com SameSite=Strict e Secure=true).
6. **Claim `tenant_id`** é incluído no principal autenticado para o `Sindico` (amarrado ao seu condomínio); para `Operator` não há claim de tenant — o tenant-alvo de cada operação é passado explicitamente no endpoint e validado.

## Alternatives Considered

### Alternative 1: SPA único com rotas `/admin/*`

- **Description**: um único bundle React serve síndico e operador, distinguindo por rota + claim.
- **Pros**: um único pipeline de build/deploy; reaproveitamento de componentes.
- **Cons**: bundle do síndico (mais exposto) carrega código sensível de backoffice; qualquer bug de roteamento/autorização vaza views; difícil aplicar CSP e isolamento de assets; atacante de conta síndico teria superfície de análise maior.
- **Why rejected**: o risco de acoplamento entre os dois públicos superou a economia de ter um único deploy.

### Alternative 2: Backoffice em Razor Pages no ASP.NET

- **Description**: páginas server-rendered servidas diretamente pela API .NET.
- **Pros**: implementação mais rápida no piloto; sem React no backoffice.
- **Cons**: cria duas stacks de UI no produto (React + Razor); destoa do padrão declarado no vision; empobrece UX do operador no médio prazo.
- **Why rejected**: incoerência arquitetural.

### Alternative 3: SSO externo (Google Workspace OIDC) apenas para operador

- **Description**: operador loga via Google Workspace; síndico com senha local.
- **Pros**: reduz gestão de senha de operador; se a equipe já usa Google Workspace, é o fluxo mais natural.
- **Cons**: adiciona integração OIDC externa no MVP; exige configurar client OAuth, callback, etc. No piloto, a equipe é pequena e controlada.
- **Why rejected para MVP:** overkill para estágio atual; pode ser evoluído em Fase 2 adicionando um provider OIDC sobre a base Identity.

## Consequences

### Positive

- Superfícies separadas reduzem risco de vazamento de código ou funcionalidade entre contextos.
- Uma única fonte de verdade de usuários no backend (ASP.NET Identity) simplifica auditoria, reset de senha e políticas comuns (hash, expiração).
- Roles deixam o modelo extensível (ex.: adicionar `Administradora` na Fase 3 sem refactor).

### Negative

- Dois pipelines de build/deploy para manter (trade-off esperado e justificável).
- Operadores e síndicos passam por telas de login distintas — é preciso deixar claro nas comunicações qual URL é qual.

### Risks

- **Risco:** operador acessa indevidamente tenant onde não deveria (ex.: com acesso futuro fatiado por "região"). **Mitigação:** no MVP, role `Operator` tem acesso global; controles finos ficam para evolução em ADR futuro se necessário.
- **Risco:** cookie auth em SPA React exige configuração cuidadosa de CORS/CSRF. **Mitigação:** mesma origin (via reverse proxy) ou CORS restrito + anti-forgery token para operações de escrita.

## Implementation Notes

- Tabela `AspNetUsers` padrão do Identity; adicionar coluna `SindicoTenantId` (nullable FK para `condominio`).
- Policies: `RequireOperator`, `RequireSindicoOfTenant` (a última verifica que o `tenant_id` do path/body bate com o claim do usuário).
- Magic link do síndico é ortogonal ao Identity (ADR-006); após consumo, o Identity recebe a senha definida via `UserManager.AddPasswordAsync`.
