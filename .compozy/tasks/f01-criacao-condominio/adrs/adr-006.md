# ADR-006: Magic Link com Token Opaco (SHA-256) em Tabela Dedicada, TTL 72h e Uso Único

## Status

Accepted

## Date

2026-04-17

## Context

F01 dispara um magic link para o primeiro síndico definir sua senha sem trafegar credenciais em canais manuais (CF3 do PRD). A política precisa cobrir geração, entrega, consumo, expiração, revogação, reemissão e auditoria. O mecanismo também precisa coexistir com o ASP.NET Core Identity (ADR-005), que administra o `PasswordHash` final.

## Decision

- **Geração**: `RandomNumberGenerator.GetBytes(32)` produz 32 bytes de entropia; codificamos em Base64URL (43 caracteres) e construímos o link `https://app.logportaria.com/setup-password?token={BASE64URL}`.
- **Persistência**: tabela global `magic_link` (não-tenant) com campos: `id` (UUID), `user_id` (FK `AspNetUsers`), `token_hash` (SHA-256 em hex, coluna única indexada), `purpose` (`PasswordSetup` no MVP), `created_at`, `expires_at`, `consumed_at`, `consumed_by_ip`. **Apenas o hash é armazenado — o token em claro jamais é persistido.**
- **Expiração**: padrão 72 horas (configurável via `MagicLinkOptions:DefaultTtl`). `expires_at` é calculado no insert.
- **Uso único**: consumo válido exige `consumed_at IS NULL` E `expires_at > now()` E `token_hash = SHA256(input)`. Na validação bem-sucedida, setamos `consumed_at` em uma única transação junto da definição da senha do usuário.
- **Reemissão**: gerar novo link invalida os anteriores do mesmo usuário+purpose que ainda estejam pendentes (atualiza `consumed_at` para um valor sentinel ou adiciona coluna `invalidated_at`). Implementação: UPDATE em massa `magic_link` marcando `invalidated_at = now()` para os registros daquele `user_id` + `purpose` com `consumed_at IS NULL`.
- **Rate-limit**: máximo de N reemissões (default 5) por usuário/janela de 24h; excedido, endpoint retorna 429 com `Retry-After`.
- **Resposta do endpoint de consumo**: não revela se o token é inválido, expirado ou já usado (retorna 400 genérico com mensagem padronizada); detalhes vão para log estruturado.
- **Auditoria**: evento `magic-link.issued` e `magic-link.consumed` em log estruturado com `user_id`, `token_id`, `ip` (sem token em claro).

## Alternatives Considered

### Alternative 1: `UserManager.GeneratePasswordResetTokenAsync` do Identity

- **Description**: usar fluxo nativo do Identity (tokens data-protected).
- **Pros**: menos código novo; ciclo de vida integrado com `UserManager`.
- **Cons**: pouca granularidade de auditoria (tokens são opaque mas não persistidos no banco para auditoria própria); expiração global configurada para a Data Protection provider, não por emissão; dificulta políticas como "invalidar todos os tokens pendentes ao reemitir".
- **Why rejected**: a persistência explícita em tabela dedicada dá controle fino sobre auditoria, reemissão e revogação, com custo marginal de implementação.

### Alternative 2: JWT auto-contido com `jti` em deny-list

- **Description**: emitir JWT assinado; validar `exp` e manter lista de `jti` consumidos.
- **Pros**: sem tabela de tokens; escala horizontal sem storage compartilhado de tokens pendentes.
- **Cons**: revogação requer manter deny-list (cai no mesmo modelo de persistência de token); JWT não-criptografado leaks claims se não usar JWE; complexidade maior para ganho nulo.
- **Why rejected**: complexidade sem benefício no estágio atual.

## Consequences

### Positive

- Tokens curtos e não reveláveis em repouso (apenas hash persistido).
- Auditoria e reemissão bem-definidas.
- Endpoint de consumo mantém resposta genérica, reduzindo vazamento de informação.

### Negative

- Uma tabela adicional e um relay de auditoria novos para manter.
- Requer cuidado operacional: migração de política de expiração (ex.: reduzir TTL) precisa ser feita por flag, não por rotação do secret (não há secret).

### Risks

- **Risco:** vazamento do token via logs de servidor (URL completa). **Mitigação:** interceptor de logging que mascara `?token=` em todos os logs; política escrita; smoke test verificando que o token não aparece em log.
- **Risco:** engenharia social — atacante pede reenvio de link para e-mail de vítima. **Mitigação:** rate-limit; envio sempre para o e-mail cadastrado do usuário; alerta de segurança após X reemissões.

## Implementation Notes

- Migração inicial cria `magic_link` sem `invalidated_at`; adiciona em migração curta antes do primeiro release público. Para o MVP podemos fazer inserir já com a coluna.
- Hash: `SHA256(UTF8(token))` em hex minúsculo; armazenado como `CHAR(64)` com índice único.
- Endpoint: `POST /api/v1/auth/password-setup` `{ token, password }` → consome + define `PasswordHash` via Identity.
- Worker opcional de housekeeping remove `magic_link` consumed/expired há mais de 90 dias (não no MVP; Open Question).
