# ADR-004: Isolamento Multi-tenant via Shared Schema com `tenant_id` e EF Core Query Filters

## Status

Accepted

## Date

2026-04-17

## Context

A plataforma é multi-tenant desde o dia 1 (RN-01, `vision.md`). Todos os serviços .NET (D01, D02, D03, D05) compartilham o mesmo modelo relacional PostgreSQL e precisam de um padrão consistente de isolamento de tenant. F01 cria o tenant raiz (`Condominio`), portanto é este TechSpec que fixa o padrão para todo o backend.

Três padrões comuns foram avaliados: shared schema com coluna discriminadora, schema-per-tenant e database-per-tenant.

## Decision

Adotaremos **shared schema** com coluna `tenant_id` (UUID) em toda tabela de dados pertencente a um tenant, combinada com:

- **EF Core global query filter** aplicado por entidade tipada como `ITenantEntity`, garantindo que toda query emitida por um DbContext configurado com um `TenantId` resolvido só enxergue dados daquele tenant.
- **Índices compostos começando por `tenant_id`** em toda tabela multi-tenant, para que planos de consulta permaneçam seletivos.
- **`TenantContext` ambíguo por request**, resolvido por um middleware ASP.NET Core que lê o tenant do claim JWT (síndico) ou do parâmetro explícito da operação de backoffice (operador gerenciando múltiplos tenants).
- **Entidades globais (usuários, tenants, logs de auditoria de plataforma)** ficam sem `tenant_id` e são acessadas via DbContext sem filter, com controle explícito no código.
- **Row-Level Security nativa do Postgres** não é habilitada no MVP (adiada para hardening pós-piloto); a defesa principal fica no EF Core query filter.

## Alternatives Considered

### Alternative 1: Schema-per-tenant

- **Description**: cada condomínio recebe um schema Postgres próprio; EF Core alterna `search_path` por request.
- **Pros**: isolamento lógico forte; restore seletivo por tenant; migrações por tenant permitem rollouts escalonados.
- **Cons**: migrações precisam rodar N vezes (uma por schema); criação de tenant fica mais cara (criar schema + rodar migração); ferramental EF Core menos natural para multi-schema; `information_schema` cresce proporcional a N tenants.
- **Why rejected**: complexidade operacional alta para os poucos tenants previstos no piloto; shared schema cobre as necessidades do MVP com menor custo.

### Alternative 2: Database-per-tenant

- **Description**: um banco Postgres para cada condomínio.
- **Pros**: isolamento máximo; compliance mais fácil de demonstrar.
- **Cons**: custo de infra linear; pool de conexões explode; upgrades e monitoramento multiplicam.
- **Why rejected**: inviável para o estágio do produto.

## Consequences

### Positive

- Padrão único e previsível em todos os serviços .NET.
- Consultas continuam em um único banco, com JOINs naturais entre entidades do mesmo tenant.
- Criação de tenant é O(1) (uma linha em `Condominio`); não há provisionamento adicional.

### Negative

- Um bug em query filter pode vazar dados entre tenants. Mitigação: teste de isolamento obrigatório na suite de integração (ver Testing Approach).
- Backups/restores são do banco inteiro; restaurar um tenant isoladamente exige trabalho manual. Mitigação: política de backup + script de extração por `tenant_id` quando necessário.

### Risks

- **Risco:** consulta esquecendo o filter (ex.: raw SQL sem WHERE tenant). **Mitigação:** code review obrigatório em migrations e queries; habilitar Roslyn analyzer customizado ou convenção de nomenclatura para métodos que ignoram filter (`IgnoreQueryFilters()` só em casos explícitos e documentados).
- **Risco:** crescimento desbalanceado de uma tabela por volume de um tenant. **Mitigação:** monitoramento por `tenant_id`; particionamento pode ser considerado na Fase 2 se necessário.

## Implementation Notes

- Entidades multi-tenant implementam `ITenantEntity { Guid TenantId { get; } }`.
- `AppDbContext` aplica `modelBuilder.Entity<T>().HasQueryFilter(e => e.TenantId == _tenantContext.TenantId)` para cada tipo que implementa `ITenantEntity`.
- `TenantContext` é scoped por request; tem `TenantId` nullable e um método `WithTenant(Guid id)` para operações de backoffice que precisam alternar tenant explicitamente.
- `condominio` é a única tabela cuja linha define o próprio tenant e não tem `tenant_id` como FK separada (o `id` do `condominio` *é* o `tenant_id`).
